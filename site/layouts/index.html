{{/* ── Problem ─────────────────────────────────────────────────── */}}
The problem

Most coding agents waste your model's attention.

They scan whole files or large chunks, stuff the prompt with noise, and hope the model figures out what matters. Read operations dominate token usage in typical agent sessions. The pain is sharpest on local models, but frontier models pay for it too in latency, cost, and diluted attention.

Default loop

Read everything. Hope the right details survive.

Whole-file dumps. Repeated reads of the same file. Irrelevant code crowding the window. Smaller models lose the thread; bigger models pay in tokens.

minicode loop

Navigate structurally. Keep the prompt lean.

A compact code map (signatures only) for orientation. Symbol-level tools fetch only the dependency cone the task needs. Repeated reads dedup automatically. The model spends its budget on judgment.

{{/* ── How it works ────────────────────────────────────────────── */}}
How it works

AST-indexed context optimization, end to end.

Index structure first. Retrieve narrowly. Spend the remaining budget on the parts of the job that actually need thought.

01
orientation

Build a structural map

Parse the project with the TypeScript compiler API. Extract symbols, signatures, and dependency edges — calls, references, extends, implements.

02
retrieval

Inject a compact code map

A signature-only skeleton of the project lands in the system prompt, ranked by reference count and entry-point heuristics. The agent orients without reading files.

03
navigation

Walk the graph

read_symbol fetches a body with its referenced types. find_references and get_dependencies follow edges. The agent moves by relationships, not by linear scrolling.

04
reasoning

Preserve the budget

File-read dedup, focus-adaptive map regeneration, and 3-phase progressive trimming keep context under budget. More room for planning, editing, and validation.

{{/* ── Why it matters ──────────────────────────────────────────── */}}
Why it matters

Long context isn't a free lunch.

Even with perfect retrieval, performance degrades with input length — attention dilutes, positional biases cause mid-context drop-off, and inference latency grows linearly with token count. minicode is built around signal density, not just compression.

local models

Make smaller local models actually viable.

Smaller models have less attention capacity to spend on noise. Cutting 60–80% of context out of the prompt is the single highest-leverage intervention for getting useful output from a model running on consumer hardware.

frontier models

Cut spend without giving up quality.

Bigger windows don't make wasted tokens free. Less prompt bloat means lower latency, lower API cost, and tighter attention on the actual task — even when the model could technically fit the whole repo.

{{/* ── FAQ ─────────────────────────────────────────────────────── */}}
FAQ

Questions technical readers usually ask first.

minicode has an opinionated architecture, so it helps to answer the common comparison questions directly.

Why AST parsing instead of connecting to the TypeScript LSP?

Because the goal isn't editor intelligence. minicode needs a lightweight, portable way to extract symbols, signatures, and dependency edges that can be injected into agent context, cached, and reused inside graph-aware tools. Using the TypeScript compiler AST keeps indexing syntax-driven and self-contained — no external language server, no editor integration layer.

The tradeoff: this is not trying to replace what an LSP does. LSPs are great for editor workflows like completions, diagnostics, and refactors. minicode is focused on the agent-runtime side: compact structural context, symbol-aware retrieval, and dependency-cone navigation.

Built-in indexing covers TypeScript and JavaScript today. The broader language story is plugin-based on purpose.

How is this different from aider?

aider is centered around file-oriented coding assistance, prompts, and git-driven edit workflows. minicode overlaps with that category, but the core thesis is different: it's built around structural context optimization, symbol-level tooling, and a graph-native interface that exposes how the agent is traversing code.

The distinction is less "better or worse" and more emphasis. aider is strong as a practical editing assistant. minicode is more explicitly an experiment in giving coding agents a structural model of the codebase rather than operating mostly at the file level.

How is this different from a tree-sitter-first approach?

tree-sitter is excellent for fast, broad syntax parsing across many languages. minicode took a narrower path for TypeScript and JavaScript by using the TypeScript compiler AST to build a project symbol graph and power tools like read_symbol, find_references, and get_dependencies.

Less breadth today in exchange for richer language-specific structural navigation. For this project, depth on the primary language mattered more than broad generic parsing — and the plugin system leaves room to grow language coverage.

Why does any of this matter if frontier models already have huge context windows?

Larger windows don't make irrelevant context free. Even when a model can fit more code, bloated prompts still cost latency, money, and attention. Recent work (Lost in the Middle; Context Length Alone Hurts LLM Performance Despite Perfect Retrieval) shows degradation is inherent to the self-attention mechanism, not just distraction from irrelevant content.

The local-model origin makes the tradeoff easy to see. The same shape shows up with hosted models, just with the cost units flipped from quality to dollars and seconds.

Which models and providers does it work with?

Anthropic (Claude) directly via API key. Anything OpenAI-compatible — that includes OpenRouter (with its free-tier hosted models), LM Studio, Ollama, and most local servers. Connect via the web UI's "Connect OpenRouter" flow or set MODEL_PROVIDER + MODEL in ~/.minicode/.env.

Can other AI agents use minicode's tools?

Yes. minicode serve exposes an MCP server at /mcp, and there's a one-click Claude Code plugin (minicode plugin install) that wires up the tools, a CLAUDE.md, and four slash commands (/minicode:analyze, /minicode:focus, /minicode:symbols, /minicode:graph). Tool calls from external agents broadcast into the same web UI graph, so you can watch them work in real time.
